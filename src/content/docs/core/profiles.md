Each OVAV profile is a **complete professional configuration**: tools, workflows, quality gates, and evidence standards. Not suggestions — governed environments.

## Available profiles

| Profile | Area | Key capabilities |
|---|---|---|
| **Platform Engineering** | Runtime, Security, CLI | Go runtime, validators, install gateway, vault |
| **Digital Product Engineering** | Full-Stack, Architecture | Web dev, APIs, databases, deploy |
| **Evidence & Decision Intelligence** | Research, Benchmarks | Source verification, competitive analysis |
| **Education & Career Development** | Learning, Curriculum | Gap detection, curriculum engine, sandbox |
| **Health & Performance Science** | Nutrition, Fitness | Meal plans, exercise design, recovery |
| **Commercial & Growth Strategy** | Business, Pricing | Market analysis, GTM, monetization |
| **DevOps & Infrastructure** | CI/CD, Cloud, SRE | Docker, monitoring, deploy pipelines |
| **UI/UX Design** | Design System, Accessibility | Prototyping, research, components |

## What makes them "professional"

Each profile includes:

1. **Domain-specific tools** — actual engines, not generic chat
2. **Quality gates** — validators and checks specific to the domain
3. **Evidence standards** — audit trails, source verification
4. **Cross-profile integration** — handoff protocols between areas
5. **Multi-model support** — swap AI models without losing workflow

## Applying a profile

```bash
ovav profile apply platform-engineering
ovav profile apply digital-product-engineering
ovav profile list
```

Profiles are additive — you can apply multiple. The runtime merges configurations intelligently.

Enterprise customers can define custom profiles specific to their organization. See [Enterprise](/reference/configuration#enterprise).
